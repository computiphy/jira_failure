# JIRA Failure Intelligence Pipeline — Requirements Document

## 1. Purpose

This pipeline turns firmware test failure HTML reports into structured JIRA bug drafts using AI agent workflows.

Given an HTML test report link or file, the system will:

1. Extract failure details, test metadata, and banners from the HTML
2. Follow linked logs and extract relevant sections with timestamps
3. Derive test steps, module name, firmware version, and DUT info
4. Generate a structured JIRA bug draft
5. Create the JIRA ticket via MCP (custom MCP server — harness only for now)

The system is an **engineering assistant** — it drafts, the engineer reviews and approves.

### Agent Compatibility

The workflow files (`.md`) must work across:

- **Cursor** (primary)
- **Antigravity** (Google DeepMind)
- **Claude Code** (Anthropic)

All instructions must be written as plain markdown workflows — no platform-specific syntax. The `skill.md` follows the standard agent-workflow `.md` convention that all three platforms support.

## 4. Pipeline Steps

### Step 1 — HTML Extraction

**Input:** HTML report (URL or local file path)

**What to extract:**

- **Banners** — test suite banners that mark test boundaries and contain test step descriptions
- **Test metadata** — test name, status (pass/fail/error/skip), duration
- **Failure details** — assertion messages, error tracebacks, failure summaries
- **DUT information** — device under test identifier, serial number if available
- **Module** — the module or component being tested (from banner or test path)
- **Firmware version** — build version, firmware version strings found in the report
- **Linked log paths** — all log file links/paths referenced in the report (relative or absolute)

**Output (structured):**

```yaml
test_name: "test_upgrade_reboot_cycle"
status: "failed"
module: "upgrade_flow"
firmware_version: "v3.2.1-rc4"
dut: "DUT-07"
failure_summary: "SSH connectivity lost after reboot during upgrade validation"
test_steps:
  - "Initialize DUT connection"
  - "Start firmware upgrade"
  - "Wait for reboot"
  - "Verify management connectivity"
  - "Validate firmware version post-upgrade"
linked_logs:
  - "logs/serial.log"
  - "logs/kernel.log"
  - "logs/upgrade.log"
```

**Rules:**

- Use `helpers.py` for HTML fetching and BeautifulSoup parsing when the report is a URL
- For local files, the agent can read the HTML directly
- If a field cannot be found, set it to `"unknown"` — do not hallucinate values
- Extract ALL linked log paths, even if they appear broken

### HTML Report Structure (Important Context)

The HTML reports are generated by a **pytest-based test framework** with the following characteristics:

- **Logging:** Python `logging` module with custom log levels and custom log handlers
- **Execution hooks:** pytest-based test execution hooks generate the report structure
- **Banners:** Constructed by filtering variables for keys and joining them — these banners appear as section headers in the HTML and encode test step information, module names, and context
- **Key libraries used upstream:** `pytest`, `logging`, `inspect`, `html` (stdlib), `py.io.saferepr`

Banners are the primary structural element — they separate test phases and contain the metadata needed to reconstruct test steps. The agent must parse banner text to extract:
- Test step sequence
- Module/component being tested
- Variable key-value pairs embedded in the banner

---

### Step 2 — Linked Log Extraction

**Input:** List of linked log paths from Step 1 + the failure context

**What to do:**

1. Open each linked log file
2. Search for sections relevant to the failure (error keywords, timestamps near failure time, stack traces)
3. Extract the relevant log snippet with surrounding context (±20 lines around each match)
4. Preserve timestamps in all extracted snippets
5. Note which log file each snippet came from

**Output (structured):**

```yaml
log_evidence:
  - source: "serial.log"
    timestamp: "2026-05-24T10:22:11"
    snippet: |
      [10:22:09] SSH connection established
      [10:22:11] Connection reset by peer
      [10:22:11] SSH session terminated
      [10:22:15] Attempting reconnect...
      [10:22:45] Connection timed out

  - source: "kernel.log"
    timestamp: "2026-05-24T10:22:10"
    snippet: |
      [10:22:10] watchdog: watchdog0 expired
      [10:22:10] Kernel panic - not syncing: softlockup
```

**Rules:**

- If a linked log does not exist or is inaccessible, note it as `"[log not found: <path>]"` and continue
- Focus on error-level messages, stack traces, reboots, connection losses, and timeouts
- Do NOT include the entire log — only relevant sections
- Maximum 50 lines per snippet

---

### Step 3 — Test Context Collection

**Input:** Outputs from Steps 1 and 2

**What to compile:**

- **Test steps** — derived from HTML banners, in execution order
- **Module** — the subsystem/component under test
- **Firmware version** — exact version string
- **DUT** — device identifier
- **Timeline** — ordered list of key events with timestamps from log evidence
- **Probable failure point** — which test step failed and why (based on evidence)

This step is a synthesis — the agent reviews what was extracted and organizes it into a coherent narrative.

---

### Step 4 — JIRA Draft Generation

**Input:** All collected context from Steps 1–3

**Output — JIRA draft fields:**

| Field | Description |
|-------|-------------|
| **Title** | Concise, specific bug title (e.g., "SSH connectivity lost after reboot during upgrade on DUT-07") |
| **Description** | Structured description containing: failure summary, test steps, firmware version, module, log evidence snippets, timeline |
| **Priority** | Suggested priority (Critical / Major / Medium / Minor) based on failure severity |
| **Component** | Module or subsystem derived from test context |
| **Environment** | Firmware version, DUT info |
| **Labels** | Auto-generated labels (e.g., `auto-triaged`, `firmware`, module name) |
| **Log Attachments** | List of log snippets to attach |

**Draft format:**

```markdown
## Summary
<one-line failure description>

## Environment
- **Firmware Version:** <version>
- **DUT:** <dut_id>
- **Module:** <module>

## Test Steps
1. <step derived from banner>
2. ...
N. **[FAILED]** <step where failure occurred>

## Failure Details
<failure summary and assertion/error message>

## Relevant Log Evidence

### serial.log (10:22:09 – 10:22:45)
```
<log snippet>
```

### kernel.log (10:22:10)
```
<log snippet>
```

## Probable Cause
<brief AI assessment — clearly marked as suggestion>
```

**Rules:**

- The draft must be readable by an engineer in under 60 seconds
- All AI assessments must be clearly labeled as suggestions
- Never state unverified conclusions as facts
- Preserve original error messages exactly — do not paraphrase

---

### Step 5 — Engineer Review

Before JIRA creation, the draft is shown to the engineer who can:

- Edit any field
- Remove or add evidence
- Change priority
- Approve or reject

**No ticket is created without explicit approval.**

---

### Step 6 — JIRA Creation via MCP

Once approved, create the JIRA ticket using the JIRA MCP server.

> **Note:** The JIRA MCP server is custom and will be implemented separately. For now, the pipeline must produce a finalized draft in a structured format (JSON/YAML) that is ready to be consumed by the MCP server when it becomes available.

**When MCP is available, it will:**

- Create issue with the approved fields
- Attach log evidence
- Set component, priority, labels, and environment
- Return the created ticket key (e.g., `FW-2041`) to the engineer

**Until then, the pipeline outputs:**

- A `jira_draft.md` rendered for engineer review
- A `jira_draft.json` structured for future MCP consumption

## 5. skill.md Specification

The `skill.md` file is the master workflow invoked by the AI agent. It must contain:

1. **Trigger** — how to invoke (e.g., user provides HTML report link)
2. **Step-by-step instructions** — referencing the prompt .md files at each stage
3. **Decision points** — what to do when logs are missing, fields unknown, etc.
4. **Output format** — the final JIRA draft structure
5. **MCP call instructions** — how to call the JIRA MCP to create the ticket

The skill.md must reference the prompt files:

```text
For HTML extraction → follow prompts/html_extraction.md
For log analysis    → follow prompts/log_extraction.md
For draft writing   → follow prompts/draft_generation.md
For examples        → see prompts/examples.md
```

### Cross-Agent Compatibility Notes

- **Cursor:** Place `skill.md` in `.cursor/rules/` or invoke via reference
- **Antigravity:** Place in `.agent/workflows/` and invoke via `/skill` slash command
- **Claude Code:** Place in `.claude/` or reference directly in conversation

The workflow must use only:
- Plain markdown instructions (no YAML frontmatter beyond what the platform requires)
- Standard tool invocations (read file, run command, write file)
- No platform-specific APIs

## 6. helpers.py Specification

A single Python file providing utilities the agent can call:

```python
# Required functions:

def fetch_html(url: str) -> str:
    """Fetch HTML content from a URL. Returns raw HTML string."""

def parse_html_report(html: str) -> dict:
    """Parse HTML report using BeautifulSoup.
    Returns structured dict with test results, banners, linked logs."""

def extract_banners(html: str) -> list[str]:
    """Extract all banner/header text from the HTML report."""

def find_linked_logs(html: str, base_path: str) -> list[str]:
    """Find all log file links/paths in the HTML. Resolve relative paths."""
```

**Dependencies:** `beautifulsoup4`, `lxml`, `requests`

This is the **only** Python file in the project.

## 8. Error Handling

- Missing HTML fields → set to `"unknown"`, continue
- Inaccessible linked logs → note as missing, continue with available logs
- No failures found in report → inform engineer, do not generate draft
- MCP connection failure → show draft to engineer, skip JIRA creation

The pipeline must **never crash silently**. Always surface what failed and why.

---

## 9. Non-Goals

This system will NOT:

- Automatically create tickets without engineer approval
- Perform root-cause analysis
- Replace test frameworks or existing tooling
- Store historical data or perform similarity search (future scope)
- Require a database or web UI

---

## 10. Success Criteria

| Metric | Target |
|--------|--------|
| Draft accuracy | ≥80% of drafts require ≤3 field edits before submission |
| Time saved | Engineer spends <2 min reviewing draft vs. 10-15 min manual triage |
| Log extraction | Relevant log sections captured in ≥90% of cases where logs exist |
| False positives | <10% of drafts contain irrelevant or misleading evidence |
